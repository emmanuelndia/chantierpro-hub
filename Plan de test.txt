Plan de test
Vérifier que l’application démarre sans erreur avec les variables d’environnement renseignées.
Vérifier que GET /api/health retourne bien { status: "ok" }.
Vérifier que les routes placeholder web/mobile principales répondent correctement.
Vérifier que Prisma peut générer le client sans erreur avec PostgreSQL/Neon.
Vérifier que le manifest PWA est servi et référence bien les icônes 192x192 et 512x512.
Vérifier que lint et format passent avec la configuration stricte.

Vérifier qu’un login valide renvoie un JWT access token et pose le refresh cookie.
Vérifier qu’une route protégée sans access token renvoie 401.
Vérifier qu’une route protégée avec rôle insuffisant renvoie 403.
Vérifier qu’un compte isActive = false renvoie ACCOUNT_DISABLED.
Vérifier que 5 échecs de login en moins de 5 minutes bloquent la tentative suivante avec TOO_MANY_ATTEMPTS et retryAfterSeconds > 0.
Vérifier que /api/auth/refresh renouvelle silencieusement l’access token avec rotation du refresh token.
Vérifier que /api/auth/logout invalide le refresh token courant et supprime le cookie.
Vérifier que POST /api/users/:id/reset-password est accessible uniquement à ADMIN.
Vérifier qu’après reset admin :
le mot de passe 12345678 fonctionne
mustChangePassword = true est visible dans /login ou /me
Vérifier que POST /api/auth/push-token crée puis met à jour un enregistrement pour le même device/token.
Vérifier que le middleware web redirige :
vers /web/login si non authentifié
vers /403 si rôle insuffisant
Vérifier que le helper fetch rejoue une requête après refresh sur premier 401, puis échoue proprement si le refresh échoue.

L’erreur “admin ne peut pas se désactiver lui-même” doit être distincte et lisible, plutôt qu’un simple 403
/api/auth/me peut rester en place ; /api/users/me devient la route profil orientée gestion utilisateur
